introduction

why does this even exist

PEP 8 is a common Python style guide that focuses on readability but the way it’s written isn’t very accessible to beginners. despite that, many of its guidelines can be used in even basic projects. adhering to a style guide early on can help you establish good habits and write more readable code. these guidelines will push you to write more Pythonic code, which can save you from later growing pains as your coding matures. when i've helped folks new to Python understand how to improve their code, i realized that they probably would have been better off if they had been introduced to PEP 8 earlier - rather than after they could understand it in full.

over time, you may find strictly following PEP 8 producesless readable code. like all conventions, PEP 8 has room for rule-breaking... well, informed rule-breaking. like in every type of writing, before you can discard convention you must understand why the it exists and what you gain by going rogue. remember: the goal is to make consistent, intentional choices - not to follow rules for their own sake. settle on (and document) sensible standards that produce readable code. keeping future readers in mind is half the battle.

so don't be too stressed! and don't be worried if you can't figure out how to implement everything discussed in everything you code. writing readable code and understanding what it means to be Pythonic is a journey, not a race. it's awesome you're getting started.

let’s go!

basics

whitespace, layout, the little basics

here's the trouble... these basics are the "rules" many throw out first. i do suggest trying to implement the basic guidelines when starting, simply because it will get you used to consistency. and again, consistency is the core of readable code. if you find that a convention is not working for you, i highly suggest asking yourself why. is it that your code could be written better, or that the style convention is failing your project?

also remember that adhering to common style conventions like this can be helpful on collaborative projects. if your project is going to be opensource and you're going to accept pull requests, using a common style will make it easier to keep your code cohesive.

indenting your code

tabs or spaces?

per PEP 8, each indentation level should be 4 spaces, no tabs. as Python's syntax is whitespace dependent, being consistent on your identation style is key.

# technically you can use two spaces
def two_spaces():
  print("like this")

# but if you're inconsistent
def bad_spacing():
    print("you can't reach this")

there are a few exceptions to this, like hanging indents. when you feel up for more detail, go ahead and read the full docs. continuation lines are an example of this:

# this line was too long, so it was continued
very_long = a_very_long_function_name(
        many_long_parameters, like_so_many,
        very_very_long_parameters)

in these cases, you'll want to use an extra level of indentation, so eight spaces.

line length

speaking of...

keep all lines no longer than 79. if it's part of something longer, like a docstring or a comment (flowing text), keep it to 72 characters. in the past, backslashes were used to note continuing text, but that has fallen out of favor.

where to break

before or after operators?

it used to be convention to break after an operator for long lines. now, however, the convention follows mathematics, as it keeps the operator closer to the thing it's operating on: 

# how not to do it
total_sum = (todays_sum +
        yesterdays_sum +
        day_before_yesterdays_sum)

# how to do it
better_total_sum = (todays_sum
        + yesterdays_sum
        + day_before_yesterdays_sum)

blank lines

they make code easier to read

top-level (so not inside another function) functions and classes should have two blank lines before and after them. if they're nested or inside a class, only use one line. use blank line to denote sections within a function, to make it easier to read.

miscellaneous

lightning round!

some more quick basics, quickly:
  • imports should be on their own line, not separated by commas
  • avoid wildcard imports when possible
  • always use utf-8 encoding
  • feel free to use "string quotes" or 'string quotes', but be consistent
  • avoid extraneous whitespace. see specific examples at the full documentation.
